But when you place it next to others, when you give it context, something changes. The structure starts to feel alive. What you bring into it begins to respond.

The system doesn't choose what goes in—you do. And the combinations you make shape the outcome. In that sense, the arrangement itself becomes the argument.

And every argument has something at stake. Here, it reflects something older: the gap between what you want and what you can actually reach. That gap isn't new—it's something we've always lived with.

It's a familiar feeling. Anyone can take part—the entry point is open. But how much you can really shape the outcome depends on what you bring with you, on what you've built up over time. Access might be equal, but reach isn't.

This condition has a name: Modulo. A system that reflects it—where desire can be arranged, and accumulation becomes visible. Even then, not everything is resolved. Some things come in already ordered. The hierarchy within each arrangement was set before you arrived. Access is still open, but order was never something you could buy.

In the end, commitment is the final step. To choose is to stop leaving everything open and say: this is it. Modulo becomes what it is when you decide to stop holding it open.

Modulo is an Ethereum collection where tokens are distributed across compositional vessels that share its name. Each token is built from a single immutable seed — one number the contract computes at mint and never alters — from which its palette, speed, animation mode, and behavior layers are all derived.

The token doesn't point to an image stored somewhere else. It generates one. Every time it's called, it runs a cellular automaton — a grid of cells whose state evolves frame by frame following local rules, generating emergent visual patterns — from inside a self-contained HTML file that lives within the token itself, fully on-chain. The automaton operates in one of eight behavioral modes, each defining how cells propagate, shift, and deform.

Some tokens carry additional layers — a bitmap system that stamps text directly into the grid, a dither pattern that introduces a repeating structure across the surface, and a rect-shift system that displaces regions of the canvas at varying density and scale. What the system produces feels alive — never the same frame twice, always within the same visual language. The rules are fixed. What they generate moves.

None of this depends on an external system. The rendering engine, the animation logic, the full computational apparatus — all of it is stored in Ethereum's contract layer through EthFS.

When a token is called, the contract doesn't return a URL. It builds the entire document on the fly, injecting each token's traits directly into the HTML, and returns it as a self-contained data URI that runs in the browser without a single external request.

There's no server that can go offline. No platform that can shut it down or change the rules. A call to any token will return the same engine, the same seed, the same behavior — in a year, in twenty years, under any condition short of the network itself disappearing. Permanence here isn't a promise. It's a property of how it's built.

When the mint window closes, ownership of the contract itself is renounced. From that block onward, no one — not even its author — can change a setting, swap a renderer, or alter behavior. What was provisional becomes fixed. The transaction that records the renunciation is part of the work as much as anything visible inside it.

Each token can be placed across the shared modulos or the owner's personal workshop (modulo 00) — up to eight per modulo, drawn one at a time from the owner's inventory. A token can only occupy one modulo at a time, and only its owner can place or remove it. Where it is placed is chosen. What that choice allows is not fixed. What can be done alone does not remain so.

Inside a modulo, the system reads the combined set and produces a single animation from it. Color palettes merge into a shared pool, influenced by position — tokens with lower rank sit higher in the arrangement and pull more strongly on what appears.

When multiple animation modes are present, the system cycles between them at a fixed interval, blending from one into the next. The position each token occupies isn't chosen by the owner. It's determined by rank, fixed from the moment the token was created. What enters is chosen. Where it lands was already decided.

What you hold when you own a token is a unique seed, a defined visual behavior, and the ability to decide where it goes. The token exists on its own, as a complete on-chain object, whether it's placed or not.

To settle is to close that openness. A transaction that burns the tokens you committed to a modulo and mints a new artifact in their place — a permanent on-chain capture of the composition at that exact moment. What the arrangement was becomes fixed. What it was doing stops. What remains is a record of the moment it was chosen. What it could have been no longer applies.

Contract

Two ERC-721 contracts are deployed on Ethereum: TokenAgent holds the originals, SettleArtifact holds the composition snapshots that appear at settle. Each token is bound to a unique seed generated at mint, from which its traits are derived deterministically and cannot be changed. The rendering engine and animation logic are stored on-chain through EthFS and shared by both contracts. A call to tokenURI() assembles the complete document on the fly — injecting that token's trait data directly into the HTML — and returns a fully self-contained data:text/html;base64. No external request, no server, no dependency. The contracts handle ownership and transfer natively. They are the source of truth for what each token is and who holds it.

Interface

The web application reads ownership and trait data from the contracts — which tokens exist, who owns them, what each one carries. Owners place tokens into modulos and configure which of their traits are active: which animation mode runs, at what speed, what visual size, whether the text, dither, or rect-shift layers are on. The token defines what is possible. The owner decides what is running. Nothing is final until a token is settled. The interface is where intention takes shape.

Sync

The shared, evolving state lives in a separate real-time layer that the contract doesn't need to know about. When one owner locks a token in a modulo, every other connected viewer sees it within a fraction of a second — the lock, the address that placed it, the duration that remains. A public activity stream surfaces every lock, expiry, and settle as it happens, so the room feels populated. The contract is still the source of truth — locks expire on schedule, settles burn on-chain — but the sync layer is what makes the shared moment legible. Drafts that an owner is still composing remain private to them; only commitments propagate. When the lock window closes without a settle, the token returns to the inventory and the participant cannot re-lock in that same modulo for a short cooldown — pressure to act, not to hold.

Settle

Settling is the act of commitment. Before a token can be settled in a shared modulo, the owner locks it for a chosen window — 1, 6, or 12 hours — making it visible to everyone and freezing its traits. If the lock window passes without a settle, the token returns to the owner's inventory and that modulo briefly cools down for them — a short window where they cannot lock again in the same vessel. When the owner signs the settle transaction, their locked tokens in that modulo are burned and a new artifact is minted in return — an on-chain NFT encoding the full composition at that moment, including tokens locked by other participants. The composition evolves between settles. The personal workshop (modulo 00) settles directly without a lock, as a solo space. To settle is to produce and reduce at once.
